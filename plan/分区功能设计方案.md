# 分区（多区服）功能设计方案

> 目标：支持"开新区"。每个区的玩家/怪物数据相互独立；不同区可运行不同内容版本
> （如 1 区 70 级版本、2 区 60 级版本，v70 会新增地图/怪物/装备/成就）；新活动可
> "全区启用"或"仅某区启用"；底层代码 bug 修复两个区都有（共享一份代码）。

---

## 一、先回答：这种好设计吗？

**好，而且这是游戏行业标准的"区服"模型。** 它的核心三原则正好对应你的三个诉求：

| 你的诉求 | 对应设计 |
|---|---|
| 每个区玩家/怪物数据相互独立 | 数据按区隔离（每区独立数据库） |
| 1 区 70 级、2 区 60 级 | 每区独立"内容版本 + 配置" |
| 新活动可全区或单区 | 每区一份"已启用功能/活动"清单 |
| 底层 bug 两区都有 | 共享一份代码，按区部署 |

已与你对齐的四项关键决策：

1. **隔离方式 = 每区独立进程 + 独立 DB**（不用"单库加 region_id 列"）。
2. **账号模型 = 一个账号可在多区拥有角色**（需要跨区共享的身份层）。
3. **版本差异 = 静态内容也随版本不同**（v70 真的新增地图/怪物/装备/成就，不是只改等级上限）。
4. **选区方式 = 游戏内选区**（现有 `select_server` 死桩改造，区记在 session 里）。

**为什么"每区独立进程 + 独立 DB"是最优解（而不是单库加 region_id 列）：**
- 当前 `db` 是**进程级全局单例**（`services/__init__.py:3` `db = SQLAlchemy()`），所有模型/服务都直接 import 它。
- 世界 Boss、掉落等状态在**进程内存**里，gunicorn 必须单 worker（`mobile_console/run.py` 已用 `-w 1`）。
- 若在单库里给每张表加 `region_id` 列，要改造几乎每一个查询，且隔离弱（容易串数据）。
- 而"每区一个进程 + 一个 SQLite 文件"天然契合现有架构：**开新区 = 起一个新进程**，隔离最强、改动最小。

## 二、整体架构

```
                一个代码库（同一份 Flask 代码，bug 修复全服生效）
                          │
          ┌───────────────┼───────────────────────────────┐
          ▼               ▼                                ▼
     [区进程 1]       [区进程 2]        ...          [共享身份库 auth_data.db]
     game_data_r1.db  game_data_r2.db                 accounts 表(账号+各区角色映射)
     内容版本 v70       内容版本 v60
     端口 5001          端口 5002
          │               │
          └───────┬───────┘
                  ▼
          反向代理 nginx（统一一个域名，按 session 里的 region 路由）
```

要点：
- **一个代码库**，部署为 N 个区进程。每个区进程 = 自己的 SQLite（`game_data_r{id}.db`）+ 自己的 gunicorn（不同端口）+ 自己的区配置。
- **共享身份库 `auth_data.db`**：存账号（username / password_hash）与"账号 → 各区角色"的映射，所有区进程共用。这样"一个账号多区角色"，**登录一次**即可在选区界面看到跨区角色。
- **反向代理（nginx）** 把多个区进程统一到一个域名/主机，按 `session['region']`（或路径）路由。session 是带**固定 SECRET_KEY** 的签名 cookie（`instance/config.py` 已固定），跨进程可移植 —— 这是代理方案能成立的前提。
- **静态资源 `static/`、模板 `templates/` 全区共用**（代码层共用，符合"底层 bug 两区都有"）。

## 三、分阶段实施

### 阶段 0：基础 —— 配置 + 内容版本化
- `config.py`：
  - 引入 `SQLALCHEMY_BINDS = {'auth': '<共享 auth_data.db 的 URI>'}`。
  - 每区进程通过 `instance/config.py`（或环境变量）覆盖**默认 DB URI**，指向自己的区库。
  - 新增区相关配置项：`REGION_ID`、`CONTENT_VERSION`、`LEVEL_CAP`、`ENABLED_ACTIVITIES`。
- **数据版本化**：`data/` 下建立版本目录，如 `data/contents/v60/`、`data/contents/v70/`，把怪物/物品/技能/地点/地图/成就等按版本分目录存放；`DataService` 按 `current_app.config['CONTENT_VERSION']` 加载对应版本（保留对旧 `data/` 的兼容加载）。**新增版本 = 加一个目录 + 区配置指向它**，不动代码。
- `db.create_all()` 必须同时在 `auth` bind 与默认（区）bind 建表（现有 `app.py:275` 只建默认 bind，需补 `db.create_all(bind='auth')`）。

### 阶段 1：共享身份 —— 一个账号多区角色
- 新增 `Account` 模型（`__bind_key__ = 'auth'`，实现 `UserMixin`）：`id`、`username`、`password_hash`、`joined_regions`（JSON，记录该账号在哪些区有角色）。
- `PlayerModel` 增加 `account_id`（在区库内），与 `Account` 关联：**一个 Account 在每区对应一条 Player**。
- `login_manager.user_loader`（`app.py:33-35`）改为加载 `Account`；游戏内通过 `session['region']` + `session['player_id']` 用 `get_active_player()` 取得当前区角色。
- 建议加一个 `before_request` 把当前区角色挂到 `g.active_player`，蓝图用 `g.active_player` 取游戏数据；把现有约 24 个游戏蓝图里的 `current_user`（取游戏数据处）批量替换为 `get_active_player()`。
- `select_server` 死桩（`blueprints/auth.py:116`）改为真实选区：列出账号在各区的角色（来自共享 accounts 库的 `joined_regions），可进入已有区，或在"可用/新区"创建角色；写入 `session['region']` + `player_id` 后进入该区流程。

### 阶段 2：每区功能 / 活动开关
- 新增功能/活动注册表（`data/activities.json` 或 `services/feature_service.py`，含活动元信息）。
- 每区配置带 `ENABLED_ACTIVITIES` 列表；提供 `is_enabled(activity_id)` 读取当前区配置判断。
- 蓝图在渲染/处理前做开关校验。
- **"全区启用"** = 改共享默认模板；**"仅某区启用"** = 该区配置覆盖。

### 阶段 3：部署与运维
- `mobile_console/run.py` 的 `start_flask()`（`:505-534`）改为按 `REGIONS` 配置循环拉起 N 个 gunicorn（各端口 + 区配置）。
- 前面加 nginx，按 `session['region']` / 路径路由到对应区进程。
- **每区 DB 独立备份**；提供"开新区"手册：加区配置 + 内容版本 + 起进程 +（可选）登记到代理。

## 四、关键风险 / 坑

| 风险 | 说明 / 对策 |
|---|---|
| **session 可移植** | `instance/config.py` 的 `SECRET_KEY` 已固定 → 跨区进程可移植，**OK**。 |
| **create_all 跨 bind** | 需分别在 `auth` 与默认 bind 建表；现有手动 `ALTER TABLE`（`app.py:278-295`）要指定正确 engine。 |
| **全局 `db` 单例** | 所有模型/服务都 import 它；引入 binds 后要保证模型 `__bind_key__` 正确、原查询仍走默认 bind。 |
| **DataService 全局缓存** | 每进程只加载本区 `CONTENT_VERSION` → 进程级天然隔离，**OK**；注意缓存键不要跨版本串。 |
| **内存态（世界 Boss / 掉落）** | 每进程一份 → 天然按区隔离，**OK**；必须保持单 worker。 |
| **现有线上库迁移** | 把 `instance/game_data.db` 拆成"共享 accounts 表 + 1 区角色表"：每个 `players` 行拆出 `Account` + 1 区 `Player` + `account_regions` 记录。 |
| **current_user 批量替换** | 用 `g.active_player` 统一取数，避免遗漏导致串区。 |

---

## 五、不在本次范围 / 待定
- 跨区交互（跨区聊天 / 战斗 / 排行榜）默认不做。
- 具体 v70 内容制作（地图、怪物等）是**数据工作**，独立于框架，不在本方案内。
- nginx 代理的具体配置细节（路由/粘性会话）作为运维项单独给出。

---

## 六、一句话总结
**共享代码 + 每区独立进程与数据库 + 每区独立内容版本与功能开关 + 共享身份库实现"一账号多区角色"**。
这正是标准区服模型，既能"开新区、不同版本并行"，又能保证"底层 bug 全服生效、活动可按区开关"。
